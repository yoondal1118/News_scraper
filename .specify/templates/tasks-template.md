# 작업 목록: [FEATURE NAME]

**Input**: `/specs/[###-feature-name]/`의 설계 문서
**Prerequisites**: plan.md(필수), spec.md(필수), research.md, data-model.md, contracts/

**Tests**: 테스트는 선택이 아니라 필수다. 모든 기능은 Pytest 테스트를 먼저
작성하고 실패(RED)를 확인한 뒤 구현한다.

**Organization**: 작업은 사용자 스토리 기준으로 그룹화하여 독립 구현/테스트가
가능하도록 한다.

## 형식: `[ID] [P?] [Story] 설명`

- **[P]**: 병렬 가능(다른 파일, 의존성 없음)
- **[Story]**: 어떤 사용자 스토리인지 표시(예: US1, US2, US3)
- 설명에는 정확한 파일 경로를 포함한다

## 경로 규칙

- **단일 프로젝트**: `src/`, `tests/`
- **웹 앱**: `backend/src/`, `frontend/src/`
- **모바일**: `api/src/`, `ios/src/` 또는 `android/src/`
- 아래 경로는 단일 프로젝트 기준이므로 plan.md 구조에 맞게 조정한다

## 의존성 & 실행 순서

### 단계 의존성

- **설정(Phase 1)**: 의존성 없음
- **기반(Phase 2)**: 설정 완료 후 진행, 모든 스토리를 차단
- **사용자 스토리(Phase 3+)**: 기반 완료 후 시작
  - 팀 여건에 따라 병렬 진행 가능
  - 또는 우선순위(P1 → P2 → P3) 순으로 진행
- **마무리(최종 단계)**: 필요한 스토리 완료 후 진행

### 사용자 스토리 의존성

- **사용자 스토리 1 (P1)**: 기반 이후 시작, 다른 스토리 의존 없음
- **사용자 스토리 2 (P2)**: 기반 이후 시작, 필요 시 US1과 통합 가능
- **사용자 스토리 3 (P3)**: 기반 이후 시작, 필요 시 US1/US2와 통합 가능

### 각 사용자 스토리 내부 규칙

- Pytest 테스트를 먼저 작성하고 실패를 확인한 후 구현한다
- 모델 → 서비스 → 엔드포인트 순서
- 핵심 구현 → 통합 순서
- 스토리 완료 후 다음 우선순위로 이동

### 병렬 가능 항목

- [P] 표시된 설정 작업은 병렬 가능
- [P] 표시된 기반 작업은 병렬 가능
- 기반 완료 후 사용자 스토리는 병렬 가능
- 각 스토리의 테스트는 병렬 가능
- 스토리 내 모델 작업은 병렬 가능

## 구현 전략

### MVP 우선 (사용자 스토리 1만)

1. Phase 1 완료: 설정
2. Phase 2 완료: 기반(모든 스토리 차단)
3. Phase 3 완료: 사용자 스토리 1
4. **STOP and VALIDATE**: 사용자 스토리 1 독립 테스트
5. 필요 시 데모/배포

### 점진적 전달

1. 설정 + 기반 완료 → 기반 준비
2. 사용자 스토리 1 추가 → 독립 테스트 → 데모
3. 사용자 스토리 2 추가 → 독립 테스트 → 데모
4. 사용자 스토리 3 추가 → 독립 테스트 → 데모
5. 각 스토리는 이전 스토리를 깨지 않고 가치를 추가한다

### 병렬 팀 전략

여러 개발자가 있는 경우:

1. 팀이 설정 + 기반을 함께 완료
2. 기반 완료 후:
   - 개발자 A: 사용자 스토리 1
   - 개발자 B: 사용자 스토리 2
   - 개발자 C: 사용자 스토리 3
3. 각 스토리를 독립적으로 완료 및 통합

## 메모

- [P] 작업 = 다른 파일, 의존성 없음
- [Story] 라벨은 작업을 스토리와 연결
- 각 스토리는 독립적으로 완료/테스트 가능해야 함
- 구현 전 테스트 실패(RED) 확인 필수
- 각 작업 또는 논리적 묶음 단위로 커밋
- 각 체크포인트에서 독립 검증
- 지양: 모호한 작업, 동일 파일 충돌, 독립성을 깨는 교차 의존성
